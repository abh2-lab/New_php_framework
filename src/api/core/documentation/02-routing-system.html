Routing System

Learn how to define, organize, and manage routes in your application

What is a Route?

A route is the mapping between an HTTP request (URL + Method) and a controller action. When a client makes a request to /api/products/list, the router determines which controller method should handle that request.

Route Definition Location

  • Application Routes: Defined in src/api/routes.php (your business logic)
  • System Routes: Defined in src/api/core/SystemRoutes.php (framework tools, loaded automatically)

Basic Route Definition

Routes are defined using the router->add() method with an array of parameters.

Minimum Required Parameters

// Simple GET route
router->add([
    'method' => 'GET',              // HTTP method
    'url' => '/products/list',       // URL pattern
    'controller' => 'ProductController@list'  // Controller@method
]);

All Available Parameters

Parameter Type Required Description
method string Required HTTP method: GET, POST, PUT, DELETE
url string Required URL pattern (e.g., /products/{id})
controller string Required Controller and method: ControllerName@methodName
desc string Optional Description shown in API documentation
visible boolean Optional Show in API docs? Default: true
group string Optional Group name for documentation organization
middleware string|array Optional Middleware name(s) to apply
params array Optional Parameter documentation for API docs
before callable Optional Function to execute before controller
after Optional callable Function to execute after controller

Complete Route Example

router->add([
    'method' => 'POST',
    'url' => '/orders/create',
    'controller' => 'OrderController@create',
    'desc' => 'Create a new order with items',
    'visible' => true,
    'group' => 'Orders',
    'middleware' => 'auth',  // Require authentication
    'params' => [
        'json' => [
            'userid' => 'User ID (integer)',
            'paymentmethod' => 'Payment method: card/cash/upi',
            'items' => 'Array of items with productid and quantity',
            'notes' => 'Optional order notes'
        ]
    ]
]);

HTTP Methods

The framework supports all standard HTTP methods. Choose the appropriate method based on the operation.

GET Retrieving Data

Use for: Fetching data without modifying anything

// Get product list
router->add([
    'method' => 'GET',
    'url' => '/products/list',
    'controller' => 'ProductController@list',
    'desc' => 'Get list of products with filters',
    'group' => 'Products',
    'params' => [
        'get' => [
            'category' => 'Optional: Filter by category',
            'limit' => 'Optional: Number of products (default: 10)'
        ]
    ]
]);

// Get single product
router->add([
    'method' => 'GET',
    'url' => '/products/{id}',
    'controller' => 'ProductController@details',
    'desc' => 'Get product details by ID',
    'group' => 'Products'
]);

POST Creating Data

Use for: Creating new resources or submitting data

router->add([
    'method' => 'POST',
    'url' => '/products/create',
    'controller' => 'ProductController@create',
    'desc' => 'Create a new product',
    'group' => 'Products',
    'params' => [
        'json' => [
            'name' => 'Product name',
            'sku' => 'Product SKU (unique)',
            'price' => 'Product price',
            'stock' => 'Stock quantity',

        ]
    ]
]);

PUT Updating Data

Use for: Updating existing resources

router->add([
    'method' => 'POST',  // Using POST for simplicity (PUT requires different handling)
    'url' => '/products/update',
    'controller' => 'ProductController@update',
    'desc' => 'Update product details',
    'group' => 'Products',
    'params' => [
        'json' => [

            'name' => 'Optional: New product name',
            'price' => 'Optional: New price',
            'stock' => 'Optional: New stock quantity'
        ]
    ]
]);

DELETE Deleting Data

Use for: Removing resources

router->add([
    'method' => 'POST',  // Using POST for simplicity
    'url' => '/orders/cancel',

Route Parameters

Routes can accept parameters in three ways: URL parameters, query strings, and request body.

1. URL Parameters

Define dynamic segments in the URL using {parameterName} syntax.

// Single URL parameter
router->add([
    'method' => 'GET',
    'url' => '/products/{id}',
    'controller' => 'ProductController@details',
    'desc' => 'Get product by ID',
    'params' => [
        'url' => [
            'id' => 'Product ID (integer)'
        ]
    ]
]);

// Multiple URL parameters
router->add([
    'method' => 'GET',
    'url' => '/categories/{categoryId}/products/{productId}',
    'controller' => 'ProductController@getByCategory',
    'params' => [
        'url' => [
            'categoryId' => 'Category ID',
            'productId' => 'Product ID'
        ]
    ]
]);

Accessing in Controller:

public function details($id) {
    // $id is automatically passed as method parameter
    $product = $this->productService->getProduct($id);
    $this->sendSuccess('Product retrieved', $product);
}

2. Query String Parameters

Parameters passed after ? in the URL: /products/list?category=electronics&limit=20

router->add([
    'method' => 'GET',
    'url' => '/products/list',
    'controller' => 'ProductController@list',
    'params' => [
        'get' => [
            'category' => 'Optional: Filter by category',
            'limit' => 'Optional: Number of results (default: 10)',
            'sort' => 'Optional: Sort order (asc/desc)'
        ]
    ]
]);

Accessing in Controller:

public function list() {
    $filters = $this->getQueryParams();  // From BaseController
    // $filters = ['category' => 'electronics', 'limit' => '20']
    
    $products = $this->productService->getProducts($filters);
    $this->sendSuccess('Products retrieved', $products);
}

3. Request Body Parameters

For POST requests, data is sent in the request body (form data or JSON).

router->add([
    'method' => 'POST',
    'url' => '/orders/create',
    'controller' => 'OrderController@create',
    'params' => [
        'json' => [  // JSON body
            'userid' => 'User ID (integer)',
            'paymentmethod' => 'Payment method: card/cash/upi',
            'items' => 'Array of items [{productid, quantity}]',
            'notes' => 'Optional order notes'
        ]
    ]
]);

Accessing in Controller:

public function create() {
    $data = $this->getRequestData();  // From BaseController
    // $data = ['userid' => 123, 'paymentmethod' => 'card', 'items' => [...]]
    
    $order = $this->orderService->createOrder($data, $data['items']);
    $this->sendSuccess('Order created', $order);
}

Route Grouping

Group related routes together to share common attributes (prefix, middleware, filters).

Basic Group with Prefix

// Group all admin routes with /admin prefix
router->group(['prefix' => '/admin'], function($router) {
    
    router->add([
        'method' => 'GET',
        'url' => '/users',  // Becomes: /admin/users
        'controller' => 'AdminController@listUsers'
    ]);
    
    router->add([
        'method' => 'POST',
        'url' => '/users/create',  // Becomes: /admin/users/create
        'controller' => 'AdminController@createUser'
    ]);
    
});

Group with Shared Middleware

// All routes in this group require authentication
router->group(['prefix' => '/orders', 'middleware' => 'auth'], function($router) {
    
    router->add([
        'method' => 'POST',
        'url' => '/create',  // Becomes: /orders/create
        'controller' => 'OrderController@create',
        'group' => 'Orders'
    ]);
    
    router->add([
        'method' => 'GET',
        'url' => '/details',  // Becomes: /orders/details
        'controller' => 'OrderController@details',
        'group' => 'Orders'
    ]);
    
});

Group with Before/After Filters

router->group([
    'prefix' => '/api/v2',
    'before' => function() {
        error_log('API v2 request started');
    },
    'after' => function($response) {
        error_log('API v2 request completed');
        return $response;
    }
], function($router) {
    
    router->add([
        'method' => 'GET',
        'url' => '/products',
        'controller' => 'ProductController@list'
    ]);
    
});

Nested Groups

// Outer group: /admin prefix
router->group(['prefix' => '/admin', 'middleware' => 'auth'], function($router) {
    
    // Inner group: /admin/products prefix
    router->group(['prefix' => '/products'], function($router) {
        
        router->add([
            'method' => 'GET',
            'url' => '/list',  // Becomes: /admin/products/list
            'controller' => 'AdminProductController@list'
        ]);
        
        router->add([
            'method' => 'POST',
            'url' => '/create',  // Becomes: /admin/products/create
            'controller' => 'AdminProductController@create'
        ]);
        
    });
    
});

Middleware Integration

Middleware allows you to filter HTTP requests before they reach your controllers or after they leave.

Important: Middleware Types in This Framework

  • Framework-Provided: BaseMiddleware, LogMiddleware, MonitoringMiddleware
  • Example Middleware: TestMiddleware, BlockingMiddleware, AuthMiddleware, AdminMiddleware
  • You Must Create: Your own AuthMiddleware, AdminMiddleware for production use

Step 1: Register Middleware

Before using middleware in routes, register it with a name.

// In routes.php (at the top)

// Register your custom middleware
router->registerMiddleware('auth', AuthMiddleware::class);
router->registerMiddleware('admin', AdminMiddleware::class);
router->registerMiddleware('log', LogMiddleware::class);

// Register example/test middleware
router->registerMiddleware('test', TestMiddleware::class);
router->registerMiddleware('blocking', BlockingMiddleware::class);

Step 2: Apply Middleware to Routes

Single Middleware

router->add([
    'method' => 'POST',
    'url' => '/orders/create',
    'controller' => 'OrderController@create',
    'middleware' => 'auth'  // Single middleware
]);

Multiple Middleware

router->add([
    'method' => 'POST',
    'url' => '/admin/users/delete',
    'controller' => 'AdminController@deleteUser',
    'middleware' => ['auth', 'admin']  // Array of middleware
]);

Middleware in Groups

// All routes in this group have 'auth' middleware
router->group(['prefix' => '/dashboard', 'middleware' => 'auth'], function($router) {
    
    router->add([
        'method' => 'GET',
        'url' => '/profile',
        'controller' => 'DashboardController@profile'
        // Automatically has 'auth' middleware
    ]);
    
    router->add([
        'method' => 'GET',
        'url' => '/orders',
        'controller' => 'DashboardController@orders',
        'middleware' => 'log'  // This route has BOTH 'auth' (from group) AND 'log'
    ]);
    
});

Middleware Execution Order

Before Request (before controller)

  1. Group middleware (outer to inner)
  2. Route-specific middleware
  3. Before filters (callables)

After Response (after controller)

  1. After filters (callables)
  2. Route-specific middleware (reverse order)
  3. Group middleware (inner to outer, reverse)

Creating Custom Middleware

// src/api/core/Middlewares/AuthMiddleware.php

namespace App\Core\Middlewares;

class AuthMiddleware extends BaseMiddleware
{
    public function before(): mixed
    {
        // Check if user is authenticated
        $token = $_SERVER['HTTP_AUTHORIZATION'] ?? null;
        
        if (empty($token)) {
            $this->halt(401, 'Unauthorized: Token required');
        }
        
        // Validate token logic here...
        
        return null;  // Continue to controller
    }
    
    public function after(mixed $response): mixed
    {
        // Optionally modify response after controller
        return $response;
    }
}

RESTful Route Naming Conventions

Action Method URL Pattern Controller Method
List all GET /products/list list() or index()
Get one GET /products/{id} details() or show()
Create POST /products/create create() or store()
Update POST /products/update update()
Delete POST /products/delete delete() or destroy()

Route Visibility in Documentation

// Visible in API docs (default)
router->add([
    'method' => 'GET',
    'url' => '/products/list',
    'controller' => 'ProductController@list',
    'visible' => true  // Will appear in /api/docs
]);

// Hidden from API docs (internal routes)
router->add([
    'method' => 'POST',
    'url' => '/internal/cleanup',
    'controller' => 'InternalController@cleanup',
    'visible' => false  // Won't appear in /api/docs
]);

System Routes vs Application Routes

How Routes are Loaded

// In src/api/index.php

// 1. System routes loaded first (framework tools)
router->setRouteType('system');
require_once __DIR__ . '/core/SystemRoutes.php';

// 2. Application routes loaded second (your business logic)
router->setRouteType('app');
require_once __DIR__ . '/routes.php';

System Routes

File: src/api/core/SystemRoutes.php

Purpose: Framework development tools

Examples:

  • /api/docs - Documentation
  • /api/service-test - Service tester
  • /api/monitoring - Performance monitoring

Flag: issystemroute = true

Application Routes

File: src/api/routes.php

Purpose: Your business logic

Examples:

  • /api/orders/create - Create order
  • /api/products/list - List products
  • /api/users/profile - User profile

Flag: issystemroute = false

Why This Matters: The monitoring dashboard separates system route performance from application route performance, giving you accurate insights into your actual business logic performance.

Route Matching & Priority

How the Router Matches Requests

  1. Method Match: HTTP method must match exactly (GET, POST, etc.)
  2. Pattern Match: URL must match the route pattern (including parameters)
  3. Specificity: More specific routes are matched first

Route Specificity Calculation

// Route specificity (higher = more specific)

// Score: 20 (two exact segments)
'/products/list'  

// Score: 15 (one exact + one parameter)
'/products/{id}'  

// Score: 10 (one exact segment)
'/products'  

// Score: 5 (one parameter)
'/{id}'  

// Score: 0 (wildcard)
'/*'

Avoid Route Conflicts

// ❌ BAD: These routes conflict!
router->add([
    'method' => 'GET',
    'url' => '/products/{id}',  // Matches: /products/123
    'controller' => 'ProductController@details'
]);

router->add([
    'method' => 'GET',
    'url' => '/products/featured',  // Also matches: /products/123
    'controller' => 'ProductController@featured'
]);

// ✓ GOOD: Make specific routes more specific
router->add([
    'method' => 'GET',
    'url' => '/products/featured',  // Register this FIRST (higher specificity)
    'controller' => 'ProductController@featured'
]);

router->add([
    'method' => 'GET',
    'url' => '/products/{id}',  // Register this SECOND
    'controller' => 'ProductController@details'
]);

Router Advanced Features

Wildcard Routes

// Trailing wildcard - matches /files/docs/guide.pdf, /files/images/logo.png, etc.
router->add([
    'method' => 'GET',
    'url' => '/files/*',
    'controller' => 'FileController@serve'
]);

// Access matched wildcard in controller
public function serve($path) {
    // $path = "docs/guide.pdf" or "images/logo.png"
}

Base Path Handling

// Framework automatically handles base path /api
// You define routes without /api prefix

// Define as:
router->add([
    'method' => 'GET',
    'url' => '/products/list',
    'controller' => 'ProductController@list'
]);

// Accessed at: http://yourdomain.com/api/products/list

Route Normalization

The framework automatically normalizes routes:

'/products/list' → '/products/list' ✓
'products/list' → '/products/list' ✓
'/products/list/' → '/products/list' ✓
'//products///list//' → '/products/list' ✓

Best Practices Summary

✓ DO

Use route groups to organize related routes
Document parameters in the params array for API docs
Apply middleware at the group level when possible
Use descriptive route names: /products/list not /prod/l
Follow RESTful conventions: GET for read, POST for write
Register more specific routes before generic ones
Use the group parameter to organize docs by feature

✗ DON'T

Define conflicting route patterns
Hardcode the /api prefix in route URLs
Mix authentication logic into route definitions (use middleware)
Create overly complex URL patterns
Forget to register middleware before using it
Use GET requests for data modification operations
