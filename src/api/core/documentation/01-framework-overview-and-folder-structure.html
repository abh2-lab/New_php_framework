Framework Overview & Folder Structure

A comprehensive guide to understanding the architecture, components, and organization of this custom PHP framework

Framework Philosophy

This framework is built on the principle of strict separation of concerns and follows a layered architecture pattern. It enforces clean code practices by separating business logic, data access, and presentation layers into distinct components.

Core Principles

  • Controller-Service-Repository Pattern: Clear separation between request handling, business logic, and data access
  • System vs Application Routes: Framework tools are isolated from your business logic
  • Convention over Configuration: Predictable file structure and naming conventions
  • Development Tools First: Built-in testing, monitoring, and debugging capabilities

System Routes vs Application Routes

The framework maintains a clear distinction between framework tools (system routes) and your business logic (application routes).

System Routes Framework Tools

These are built-in development and monitoring tools provided by the framework. They are defined in src/api/core/SystemRoutes.php and are automatically loaded before your application routes.

Important: System routes are for development/monitoring purposes. In production, you may want to restrict access to these routes using middleware or environment-based controls.
Route Purpose When to Use
/api/docs Interactive API documentation Exploring and testing your API endpoints
/api/service-test Service layer testing tool Testing business logic methods directly
/api/repository-test Repository layer testing tool Testing database queries in isolation
/api/monitoring Real-time performance dashboard Analyzing performance and bottlenecks
/api/runmigration Database migration runner Applying SQL migrations to database
/curl_runner2.html Visual HTTP request tool Testing APIs without command line

Application Routes Your Business Logic

These are your custom routes that handle your application's business requirements. They are defined in src/api/routes.php and represent your actual API endpoints.

// Example from routes.php
router->add([
    'method' => 'POST',
    'url' => '/orders/create',
    'controller' => 'OrderController@create',
    'desc' => 'Create a new order with items',
    'visible' => true,
    'group' => 'Orders'
]);

router->add([
    'method' => 'GET',
    'url' => '/products/list',
    'controller' => 'ProductController@list',
    'desc' => 'Get list of products',
    'visible' => true,
    'group' => 'Products'
]);

Built-in Development Tools

The framework comes with powerful development tools that accelerate your workflow and make debugging easier.

📚 API Documentation Viewer

/api/docs
Development Tool

Purpose: Interactive, auto-generated API documentation with live testing capabilities.

Key Features:

  • Automatically scans routes.php and displays all registered routes
  • Grouped by categories (Orders, Products, etc.)
  • Live API testing directly from the browser
  • Auto-fill feature generates dummy data for testing
  • Dark mode support with theme persistence
  • Search functionality across all endpoints
  • Environment variable management (CRUD operations)
  • Syntax highlighting for JSON responses

When to Use:

  • Exploring available API endpoints
  • Testing endpoints without Postman
  • Sharing API documentation with frontend developers
  • Quick environment variable changes during development
Pro Tip: The docs controller automatically reads parameter definitions from your routes, so always document your route parameters for better API documentation.

🔧 Service Tester

/api/service-test
Development Tool

Purpose: Test service layer methods independently without going through controllers or HTTP requests.

Key Features:

  • Uses PHP Reflection to discover all service classes in src/api/services/
  • Lists all public methods with parameter information
  • JSON-based argument input for method testing
  • Real-time execution with error handling
  • Debug output shows return values and exceptions
  • Dark-themed professional UI

When to Use:

  • Testing business logic in isolation
  • Debugging service methods without HTTP overhead
  • Verifying complex business rules
  • Testing transaction handling
Example Use Case: Test your OrderService::createOrder() method with different data combinations to verify validation logic before writing controller code.

💾 Repository Tester

/api/repository-test
Development Tool

Purpose: Test repository layer methods and database queries directly without services or controllers.

Key Features:

  • Auto-discovers all repository classes in src/api/repositories/
  • Reflects all public methods with their signatures
  • Auto-argument generation for quick testing
  • JSON argument support for complex parameters
  • Real-time query execution and result display
  • Exception handling with detailed error messages

When to Use:

  • Testing database queries in isolation
  • Verifying SQL logic without service layer
  • Debugging data retrieval issues
  • Testing repository methods with edge cases
Example Use Case: Test your ProductRepository::findById() method with various product IDs to ensure it handles missing records correctly.

🌐 Curl Runner (GUI)

/curl_runner2.html
Development Tool

Purpose: Visual HTTP request tool for testing APIs without command-line curl or Postman.

Key Features:

  • GUI-based HTTP request builder
  • Support for GET, POST, PUT, DELETE methods
  • Custom headers and request body
  • Formatted response display
  • No installation required

When to Use:

  • Quick API endpoint testing
  • Testing external APIs
  • When you don't want to use command-line tools

📊 Monitoring Dashboard

/api/monitoring
Performance

Purpose: Real-time application performance monitoring with live charts and metrics.

Key Features:

  • Real-time SSE Streaming: Server-Sent Events push live metrics to the dashboard
  • Memory Usage Tracking: Current, peak, and system route average memory consumption
  • Response Time Analysis: P50 (median) and P95 (95th percentile) metrics for app and system routes
  • Requests Per Second (RPS): Live request rate tracking
  • Database Query Metrics: Query count, execution time, and percentage of total response time
  • Active Requests Counter: Shows concurrent request handling
  • Top Slowest Endpoints: Identifies performance bottlenecks
  • Health Checks: Database connectivity, disk usage, PHP version
  • Historical Data: View metrics by date or date range
  • System vs App Separation: Tracks framework tools separately from your business logic
  • Color-coded Indicators: Green (good), yellow (warning), red (critical)

Technical Implementation:

  • Uses MonitoringMiddleware to capture metrics on every request
  • Logs stored in src/api/logs/monitoring/ directory (JSON format)
  • Non-blocking logging (happens after response is sent)
  • Chart.js for live data visualization
  • Supports multiple concurrent viewers (configurable limit)

Metrics Captured Per Request:

  • Response time (total request duration)
  • CPU time (actual processing time)
  • Memory used (RAM consumed)
  • Peak memory (highest memory point)
  • Database queries count
  • Database total execution time
  • Database time as percentage of response time
  • HTTP status code
  • Request method and endpoint
  • Client IP and User-Agent
  • Timestamp

When to Use:

  • Identifying slow endpoints
  • Detecting memory leaks
  • Finding N+1 query problems (high query count)
  • Analyzing database bottlenecks
  • Monitoring production performance
  • Load testing analysis
Performance Impact: Monitoring middleware adds minimal overhead (~2-5ms) since logging happens after the response is sent. In production, consider enabling only when needed or restricting access.
Color Coding Guide:
  • Green (Efficient): Memory <50MB, Response <200ms, DB Queries <10
  • Yellow (Warning): Memory 50-100MB, Response 200-500ms, DB Queries 10-50
  • Red (Critical): Memory >100MB, Response >500ms, DB Queries >50

🗄️ Migration Runner

POST /api/runmigration
Database

Purpose: Execute SQL migration files to set up or update your database schema.

Key Features:

  • Automatically scans src/api/databases/*.sql files
  • Executes migrations in natural order (001_init.sql, 002_orders.sql, etc.)
  • Handles foreign key constraints (disables/re-enables automatically)
  • Idempotent execution (safe to run multiple times)
  • Detailed execution report with statement counts
  • Error handling per statement

When to Use:

  • Initial database setup
  • Applying schema changes
  • Resetting development database
Pro Tip: Name your migration files with numeric prefixes (001_, 002_) to control execution order.

Layer Responsibilities

The framework enforces strict separation of concerns across three primary layers:

Controllers (Request Layer)

Responsibility: Handle HTTP requests, validate input, call services, return responses.

What Controllers SHOULD do:

  • Receive and parse HTTP requests
  • Validate request data (using validateRequest())
  • Call appropriate service methods
  • Format and return JSON responses
  • Handle exceptions and return appropriate HTTP status codes

What Controllers SHOULD NOT do:

  • ❌ Contain business logic
  • ❌ Directly query the database
  • ❌ Process data transformations

Services (Business Logic Layer)

Responsibility: Implement business rules, orchestrate data operations, handle transactions.

What Services SHOULD do:

  • Implement business rules and validation
  • Orchestrate multiple repository calls
  • Handle database transactions
  • Transform and process data
  • Throw business logic exceptions

What Services SHOULD NOT do:

  • ❌ Parse HTTP requests
  • ❌ Format HTTP responses
  • ❌ Write SQL queries directly

Repositories (Data Access Layer)

Responsibility: Execute database queries, map data to objects.

What Repositories SHOULD do:

  • Perform database CRUD operations
  • Write optimized SQL queries
  • Map database results to arrays
  • Handle database-specific exceptions

What Repositories SHOULD NOT do:

  • ❌ Implement business logic
  • ❌ Handle transactions (leave to services)
  • ❌ Format HTTP responses

ENABLE_MONITORING=true

RATE_LIMIT_REQUESTS=100

Security Note: Never commit your .env file to version control. Use .env.example as a template.

Getting Started

Now that you understand the framework's structure and tools, here's how to start building your application:

  1. Visit the Documentation: Open /api/docs to see all available routes
  2. Run Migrations: POST to /api/runmigration to set up your database
  3. Create Your First Route: Add routes in src/api/routes.php
  4. Build Your Controller: Create a controller in src/api/controllers/
  5. Implement Business Logic: Create a service in src/api/services/
  6. Add Data Access: Create a repository in src/api/repositories/
  7. Test Your API: Use /api/docs to test your endpoints
  8. Monitor Performance: Check /api/monitoring for performance metrics
