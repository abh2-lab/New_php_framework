Writing Migration Files

Migrations are SQL files that define your database schema.

This framework uses a simple, file-based migration system where you write raw SQL files that are executed in order to create tables, indexes, foreign keys, and seed data.

What are Migrations?

Migrations are version-controlled SQL scripts that:

  • Define database structure: Tables, columns, indexes, constraints
  • Are idempotent: Can be run multiple times safely without errors
  • Are ordered: Execute in a predictable sequence
  • Track changes: Provide a history of schema evolution
  • Enable collaboration: Team members share the same schema

Migration Folder Structure

All migration files are stored in:

src/api/databases/ ├── 001_init.sql ├── 002_orders.sql ├── 003_users.sql └── 004_categories.sql

File Naming Convention

REQUIRED

Migration files must follow this pattern:

-- Pattern: {number}_{description}.sql

-- Examples:
001_init.sql
002_orders.sql
003_users.sql
100_add_indexes.sql

Important: Files are executed in natural sort order (001, 002, 003...). The numeric prefix determines execution order.

The Migration Runner: InitController

The framework provides an InitController that handles migration execution:

Endpoint

// Route: POST
/api/init/migrate

// Controller:
App\Controllers\InitController@migrate

How It Works

  1. Scans src/api/databases/ for all .sql files
  2. Sorts files naturally (001, 002, 003...)
  3. Disables foreign key checks: SET FOREIGN_KEY_CHECKS=0
  4. Executes each file's SQL statements sequentially
  5. Re-enables foreign key checks: SET FOREIGN_KEY_CHECKS=1
  6. Returns success/error report

Response Format

// Success response:
{
    "success": true,
    "message": "All migrations executed successfully",
    "directory": "/path/to/databases",
    "files_found": 3,
    "total_statements_found": 12,
    "total_statements_executed": 12,
    "files": [
        {
            "file": "001_init.sql",
            "success": true,
            "statements_found": 5,
            "statements_executed": 5,
            "errors": []
        }
    ]
}

Writing Migration Files

Basic Structure

SQL 
-- ============================================
-- Migration: Order System Tables
-- File: 002_orders.sql
-- Purpose: Create orders, order_items tables
-- Version: 1.0
-- Date: 2026-01-03
-- ============================================

-- Create orders table
CREATE TABLE IF NOT EXISTS orders (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id INT UNSIGNED NOT NULL,
    total_amount DECIMAL(10,2) NOT NULL,
    status ENUM('pending', 'confirmed', 'cancelled') DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_user_id (user_id),
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

Key SQL Patterns

1. CREATE TABLE IF NOT EXISTS

IDEMPOTENT

Always use IF NOT EXISTS to make migrations idempotent:

CREATE TABLE IF NOT EXISTS products (
    -- columns...
);

Why? This allows you to run migrations multiple times without getting "table already exists" errors.

2. Column Definitions

Data Type Use Case Example
INT UNSIGNED IDs, positive integers id INT UNSIGNED AUTO_INCREMENT
VARCHAR(n) Short text (names, SKUs) name VARCHAR(255) NOT NULL
TEXT Long text (descriptions) description TEXT
DECIMAL(10,2) Prices, amounts price DECIMAL(10,2) NOT NULL
ENUM Fixed status values status ENUM('active', 'inactive')
TIMESTAMP Dates and times created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP

3. Indexes

PERFORMANCE 
-- Add indexes for frequently queried columns
INDEX idx_user_id (user_id),
INDEX idx_status (status),
INDEX idx_created_at (created_at),

-- Unique index for unique values
UNIQUE INDEX idx_sku (sku)

4. Foreign Keys

RELATIONSHIPS 
-- ON DELETE CASCADE: Delete child records when parent is deleted
FOREIGN KEY (order_id) REFERENCES orders(id) ON DELETE CASCADE,

-- ON DELETE RESTRICT: Prevent deletion if child records exist
FOREIGN KEY (product_id) REFERENCES products(id) ON DELETE RESTRICT

5. Timestamps (Auto-Management)

-- created_at: Set once on insert
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

-- updated_at: Auto-update on every modification
updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

-- Nullable timestamp for optional events
cancelled_at TIMESTAMP NULL

6. Engine and Charset

REQUIRED 
-- Always specify at the end of CREATE TABLE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

Important: Always use InnoDB (supports transactions and foreign keys) and utf8mb4 (full Unicode support including emojis).

Real Framework Example

Here's the actual orders_system_migration.sql from the framework:

-- ============================================
-- Orders System Migration
-- Purpose: Create tables for Products, Orders, Order Items, and Transactions
-- Version: 1.0
-- Date: 2025-12-30
-- ============================================

-- PRODUCTS TABLE
CREATE TABLE IF NOT EXISTS products (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    sku VARCHAR(100) UNIQUE NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    stock INT NOT NULL DEFAULT 0,
    category VARCHAR(100),
    description TEXT,
    status ENUM('active', 'inactive') DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_sku (sku),
    INDEX idx_category (category),
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- ORDERS TABLE
CREATE TABLE IF NOT EXISTS orders (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id INT UNSIGNED NOT NULL,
    total_amount DECIMAL(10,2) NOT NULL,
    payment_method ENUM('card', 'cash', 'upi') NOT NULL,
    status ENUM('pending', 'confirmed', 'cancelled', 'completed') DEFAULT 'pending',
    notes TEXT,
    cancellation_reason TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    cancelled_at TIMESTAMP NULL,
    INDEX idx_user_id (user_id),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- ORDER ITEMS TABLE
CREATE TABLE IF NOT EXISTS order_items (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    order_id INT UNSIGNED NOT NULL,
    product_id INT UNSIGNED NOT NULL,
    quantity INT NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    subtotal DECIMAL(10,2) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_order_id (order_id),
    INDEX idx_product_id (product_id),
    FOREIGN KEY (order_id) REFERENCES orders(id) ON DELETE CASCADE,
    FOREIGN KEY (product_id) REFERENCES products(id) ON DELETE RESTRICT
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- ORDER TRANSACTIONS TABLE (Audit Log)
CREATE TABLE IF NOT EXISTS order_transactions (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    order_id INT UNSIGNED NOT NULL,
    action VARCHAR(50) NOT NULL COMMENT 'order_created, order_cancelled, payment_received, etc.',
    amount DECIMAL(10,2) NOT NULL,
    payment_method VARCHAR(50),
    status VARCHAR(50) COMMENT 'success, failed, refunded, etc.',
    notes TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_order_id (order_id),
    INDEX idx_action (action),
    FOREIGN KEY (order_id) REFERENCES orders(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- SEED DATA: Sample Products
INSERT INTO products (name, sku, price, stock, category, description, status)
VALUES
    ('Laptop Pro 15', 'LAP-001', 999.99, 10, 'Electronics', 'High-performance laptop with 16GB RAM', 'active'),
    ('Wireless Mouse', 'MOU-001', 29.99, 50, 'Electronics', 'Ergonomic wireless mouse', 'active'),
    ('USB-C Cable', 'CAB-001', 9.99, 100, 'Accessories', 'High-speed USB-C cable', 'active')
ON DUPLICATE KEY UPDATE updated_at = CURRENT_TIMESTAMP;

Breaking Down This Migration

Products Table

  • Primary Key: id with auto-increment
  • Unique Constraint: sku must be unique
  • ENUM: status limited to 'active' or 'inactive'
  • Indexes: On sku, category, and status for fast lookups

Orders Table

  • DECIMAL: total_amount for precise currency values
  • ENUM: payment_method and status for controlled values
  • Nullable Timestamp: cancelled_at only set when order is cancelled

Order Items Table

  • Foreign Key to Orders: ON DELETE CASCADE (delete items when order is deleted)
  • Foreign Key to Products: ON DELETE RESTRICT (prevent product deletion if in orders)

Order Transactions Table (Audit Log)

  • Immutable Log: Records all actions on orders
  • Comments: Use COMMENT to document column purpose

Seed Data

  • ON DUPLICATE KEY UPDATE: If SKU exists, just update timestamp (idempotent)

Seed Data Pattern

IDEMPOTENT

To insert sample data safely (without errors on re-runs):

INSERT INTO products (name, sku, price, stock, category, status)
VALUES
    ('Product A', 'SKU-001', 99.99, 10, 'Category1', 'active'),
    ('Product B', 'SKU-002', 149.99, 5, 'Category2', 'active')
ON DUPLICATE KEY UPDATE 
    updated_at = CURRENT_TIMESTAMP;

Why ON DUPLICATE KEY UPDATE?

This prevents "duplicate key" errors when running migrations multiple times. If the record exists (based on unique key like SKU), it just updates the timestamp instead of failing.

Error Handling

Common Errors (Auto-Ignored)

The migration runner automatically ignores these safe errors:

Error Code Description Why Ignored
23000 Duplicate key entry Seed data already exists (ON DUPLICATE KEY UPDATE)
42S01 Table already exists CREATE TABLE IF NOT EXISTS still triggers this on re-run

Migration Best Practices

✅ DO:

  • Use IF NOT EXISTS for all CREATE TABLE statements
  • Number migration files sequentially (001, 002, 003...)
  • Add comments explaining the purpose of each table
  • Use DECIMAL for currency, never FLOAT
  • Always specify ENGINE=InnoDB and utf8mb4
  • Add indexes on foreign keys and frequently queried columns
  • Use ON DUPLICATE KEY UPDATE for seed data
  • Use TIMESTAMP DEFAULT CURRENT_TIMESTAMP for audit fields
  • Document ENUM values and their meanings
  • Use descriptive index names (idx_user_id, not idx1)

⚠️ BE CAREFUL:

  • Don't modify old migration files after they've been run
  • Don't delete migration files from the repository
  • Test migrations locally before deploying to production
  • Be cautious with ON DELETE CASCADE (can delete data unintentionally)
  • Backup your database before running migrations on production

❌ DON'T:

  • Don't use DROP TABLE in migrations (destructive)
  • Don't use FLOAT for money (precision errors)
  • Don't forget foreign key indexes (performance issue)
  • Don't use MyISAM engine (no transaction support)
  • Don't forget ON UPDATE CURRENT_TIMESTAMP for updated_at
  • Don't hardcode data that might change (use seed scripts instead)

Common Migration Patterns

Pattern 1: User Authentication Table

CREATE TABLE IF NOT EXISTS users (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    name VARCHAR(255) NOT NULL,
    role ENUM('user', 'admin') DEFAULT 'user',
    status ENUM('active', 'inactive', 'suspended') DEFAULT 'active',
    email_verified_at TIMESTAMP NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_email (email),
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

Pattern 2: Pivot Table (Many-to-Many)

CREATE TABLE IF NOT EXISTS product_categories (
    product_id INT UNSIGNED NOT NULL,
    category_id INT UNSIGNED NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (product_id, category_id),
    FOREIGN KEY (product_id) REFERENCES products(id) ON DELETE CASCADE,
    FOREIGN KEY (category_id) REFERENCES categories(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

Pattern 3: Soft Deletes

CREATE TABLE IF NOT EXISTS posts (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    content TEXT,
    deleted_at TIMESTAMP NULL,  -- NULL = active, NOT NULL = soft deleted
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_deleted_at (deleted_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

Troubleshooting

Problem: Foreign Key Constraint Fails

Solution: Ensure parent table is created before child table. Check migration file order.

Problem: Syntax Error in SQL

Solution: Test SQL in phpMyAdmin or MySQL CLI first. Check for missing semicolons and quotes.

Problem: Migration Runs but Tables Not Created

Solution: Check database connection. Verify DB_NAME in .env file.

Problem: Duplicate Key Error on Seed Data

Solution: Use ON DUPLICATE KEY UPDATE pattern shown above.
