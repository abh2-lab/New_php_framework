Database & Query Handling

This framework uses a modern PDO wrapper class called Database that provides a clean, chainable interface for all database operations.

Instead of working with raw PDO, you'll use the Database class which offers security, performance tracking, transaction support, and a simpler API.

Overview: The Database Class

The Database class is located at:

// File path
src/api/core/Database.php

// Namespace
namespace App\Core;

class Database { ... }

Key Features

  • PDO Wrapper: Modern interface over raw PDO
  • Prepared Statements: All queries use prepared statements for security
  • Transaction Support: Built-in begin/commit/rollback with automatic error handling
  • Performance Tracking: Automatic query count, execution time, and data size metrics
  • Helper Methods: Simplified select(), insert(), update(), delete()
  • Debug Mode: Logs slow queries (>500ms) when debug is enabled

How Controllers and Repositories Get Database Access

In Controllers (via BaseController)

All controllers extend BaseController, which automatically initializes the database:

// Inside BaseController constructor
protected ?Database $db = null;

public function __construct() {
    // Get PDO connection
    $pdo = DatabaseConnection::pdo();

    // Create Database wrapper instance
    $this->db = new Database($pdo);
}

Controllers then pass $this->db to repositories:

// Example: ProductController
private ProductRepository $productRepo;

public function __construct() {
    parent::__construct();
    $this->productRepo = new ProductRepository($this->db);
}

In Repositories

Repositories receive the Database instance via constructor injection:

class ProductRepository {
    private Database $db;

    public function __construct(Database $db) {
        $this->db = $db;
    }

    // Now use $this->db for all queries
}

Core Query Method: query()

The query() method is the foundation of all database operations. It accepts a configuration array:

Method Signature

public function query(array $config): array

Configuration Array Options

Key Type Required Description
query string ✅ Yes SQL query string with named placeholders
params array No Associative array of parameters (default: [])
fetchAssoc bool No Return associative array (true) or numeric (false). Default: true
withSuccess bool No Return structured response with success/id/affectedRows. Default: false
returnSql bool No Return interpolated SQL string (for debugging). Default: false

Basic Query Example

// Simple SELECT query
$products = $this->db->query([
    'query' => 'SELECT * FROM products WHERE category = :category',
    'params' => ['category' => 'Electronics']
]);

// Result: Array of product rows
// [
//   ['id' => 1, 'name' => 'Laptop', 'category' => 'Electronics'],
//   ['id' => 2, 'name' => 'Mouse', 'category' => 'Electronics']
// ]

Query with Structured Response

// INSERT with structured response
$result = $this->db->query([
    'query' => 'INSERT INTO products (name, sku, price) VALUES (:name, :sku, :price)',
    'params' => [
        'name' => 'Wireless Mouse',
        'sku' => 'MOU-002',
        'price' => 29.99
    ],
    'withSuccess' => true
]);

// Result structure:
// [
//   'success' => true,
//   'affectedRows' => 1,
//   'id' => 5,           // Last insert ID
//   'data' => []
// ]

The Database class provides convenient helper methods that wrap query():

$products = $this->db->select(

INSERT

$result = $this->db->insert(

Security: Prepared Statements

CRITICAL

NEVER concatenate user input directly into SQL queries!

Always use named placeholders (:param) and parameter binding.

✅ Correct (Secure)

// Using named placeholders
$products = $this->db->select(
    'SELECT * FROM products WHERE category = :category AND price < :max_price',
    [
        'category' => $userInput,
        'max_price' => $maxPrice
    ]
);

❌ Wrong (SQL Injection Risk)

// NEVER do this!
$query = "SELECT * FROM products WHERE category = '$userInput'";
$products = $this->db->select($query);

How It Works

Behind the scenes, the framework:

  1. Prepares the SQL statement with placeholders
  2. Binds parameters safely using PDO's binding mechanism
  3. Executes the prepared statement
  4. Returns the results

Transaction Management

TRANSACTIONS

Transactions ensure that multiple database operations either all succeed or all fail together, maintaining data consistency.

Manual Transaction Control

// Start transaction
$this->db->beginTransaction();

try {
    // Multiple operations
    $orderResult = $this->db->insert(
        'INSERT INTO orders (user_id, total_amount) VALUES (:user_id, :total)',
        ['user_id' => 10, 'total' => 150.00]
    );

    $orderId = $orderResult['id'];

    $this->db->insert(
        'INSERT INTO order_items (order_id, product_id, quantity) VALUES (:order_id, :product_id, :qty)',
        ['order_id' => $orderId, 'product_id' => 5, 'qty' => 2]
    );

    // Commit if all succeeded
    $this->db->commit();

} catch (\Exception $e) {
    // Rollback on error
    $this->db->rollback();
    throw $e;
}

Automatic Transaction (Recommended)

Use the transaction() method with a callback for automatic commit/rollback:

$result = $this->db->transaction(function($db) {
    // All operations inside this callback are in a transaction
    $orderResult = $db->insert(
        'INSERT INTO orders (user_id, total_amount) VALUES (:user_id, :total)',
        ['user_id' => 10, 'total' => 150.00]
    );

    $orderId = $orderResult['id'];

    $db->insert(
        'INSERT INTO order_items (order_id, product_id, quantity) VALUES (:order_id, :product_id, :qty)',
        ['order_id' => $orderId, 'product_id' => 5, 'qty' => 2]
    );

    return $orderId; // Return value from transaction
});

// Automatically commits on success
// Automatically rolls back on exception

Transaction Helper Methods

Method Description
beginTransaction() Start a new transaction
commit() Commit the current transaction
rollback() Rollback the current transaction
inTransaction() Check if currently inside a transaction (returns bool)
getTransactionLog() Get array of queries executed in current transaction (debug mode)
transaction(callable $callback) Execute callback with automatic commit/rollback

Database Connection Management

DatabaseConnection Singleton

The framework uses a singleton pattern for database connections via the DatabaseConnection class:

// Located at: src/api/core/config/DatabaseConnection.php

use App\Core\Config\DatabaseConnection;

// Get PDO connection (singleton)
$pdo = DatabaseConnection::pdo();

// Wrap it with Database class
$db = new Database($pdo);

Environment-Based Configuration

The connection automatically adapts based on environment variables:

1. CLI Mode (Command Line)

// Uses CLI-specific variables:
DB_HOST_CLI = 127.0.0.1
DB_PORT_CLI = 3306
DB_USER = root
DB_PASSWORD = abcd
DB_NAME = testdb

2. Development Mode

// When APP_ENV = development|dev|local:

DB_PORT = 3306
DB_USER = root
DB_PASSWORD = abcd

3. Production Mode

// When APP_ENV = production:

Automatic Timezone Setup

The connection automatically sets the timezone:

$pdo->exec("SET time_zone = 'Asia/Kolkata'");

Performance Tracking

MONITORING

The Database class automatically tracks performance metrics for every query:

Tracked Metrics

  • Query Count: Total number of queries executed ($GLOBALS['query_count'])
  • Query Time: Total execution time in milliseconds ($GLOBALS['query_time'])
  • Max Query Time: Slowest query execution time ($GLOBALS['max_query_time'])
  • Data Size: Total data returned in KB ($GLOBALS['db_data_size'])
  • Max Data Size: Largest single query result ($GLOBALS['max_db_data_size'])

Slow Query Logging

When debug mode is enabled, queries taking longer than 500ms are automatically logged:

// Logged to error_log when query > 500ms
"SLOW QUERY (753.42ms): SELECT * FROM products JOIN categories..."

Integration with Monitoring

The MonitoringMiddleware automatically captures these metrics and includes them in performance reports:

// Monitoring response includes:
{
    "db_queries": 5,
    "db_time": 42.35,           // ms
    "max_db_time": 18.20,       // ms
    "avg_db_time": 8.47,        // ms
    "db_percentage": 21.5,      // % of total response time
    "db_data_size": 124.5,     // KB
    "response_time": 197.2      // ms
}

public function findAll(int $limit = 50): array {

Example 3: OrderRepository - Find with JOIN

public function getOrderItems(int $orderId): array {
    $query = "SELECT 
                   oi.id,
                   oi.product_id,
                   p.name as product_name,
                   p.sku,
                   oi.quantity,
                   oi.price,
                   oi.subtotal
               FROM order_items oi
               INNER JOIN products p ON oi.product_id = p.id
               WHERE oi.order_id = :order_id
               ORDER BY oi.id";

    return $this->db->select($query, [
        'order_id' => $orderId
    ]);
}

Example 4: Dynamic UPDATE with Multiple Fields

// File: src/api/repositories/OrderRepository.php

public function update(int $orderId, array $data): bool {
    $setClauses = [];
    $params = ['id' => $orderId];

    // Build SET clauses dynamically
    foreach ($data as $key => $value) {
        $setClauses[] = "{$key} = :$key";
        $params[$key] = $value;
    }

    $setClauses[] = "updated_at = NOW()";

    $query = "UPDATE orders SET " . implode(', ', $setClauses) . " WHERE id = :id";

    $result = $this->db->update($query, $params);

    return $result['affectedRows'] > 0;
}

Error Handling

Database errors are automatically wrapped in DatabaseException

Catching Database Errors

use App\Core\Exceptions\DatabaseException;

try {
    $product = $this->db->first(
        'SELECT * FROM products WHERE id = :id',
        ['id' => $productId]
    );
} catch (DatabaseException $e) {
    // In production, detailed DB errors are hidden
    // In debug mode, full error is shown
    error_log("Database error: " . $e->getMessage());
    throw $e;
}

Production vs Debug Mode

  • Debug Mode (DEBUG_MODE=true): Full error messages, query logging, slow query warnings
  • Production Mode: Generic error messages, no query logging, errors are logged but not exposed

Best Practices Summary

✅ DO:

  • Always use named placeholders for parameters
  • Use first() when expecting a single result
  • Use transactions for multi-step operations
  • Prefer the transaction() callback for automatic handling
  • Use helper methods (select, insert, update, delete) for clarity
  • Handle DatabaseException in repositories or services

❌ DON'T:

  • Never concatenate user input into SQL strings
  • Don't forget to commit or rollback transactions
  • Don't use SELECT * in production (specify columns)
  • Don't expose raw database errors to end users
  • Don't perform database queries in controllers (use repositories)
